# Eyebert Technical Report

*A complete account of the project: why it exists, the theory it rests on,
how it is built, how it was verified, what failed along the way, and where it
goes next. Written to be read front-to-back as the basis for future
development.*

---

## 1. Purpose and Gap Analysis

### 1.1 The problem space

Every multi-gigabit serial link — PCIe, Ethernet, USB, the SerDes lanes
between an FPGA and anything else — degrades the signal between transmitter
and receiver. Board traces attenuate high frequencies, connectors reflect,
crosstalk injects noise. The fundamental question of link validation is:
*at this operating point, how often does a transmitted bit arrive wrong?*
That quantity is the **bit error rate (BER)**, and the instrument that
measures it is a **BERT** (bit error rate tester).

A BERT does three things: generate a known pseudo-random pattern, compare
the received stream against a locally regenerated copy of that pattern, and
count disagreements. Layered on top, two standard analyses characterize a
link: the **BER waterfall** (BER versus one swept parameter — TX swing,
equalizer gain, channel loss) and the **eye diagram** (a 2D histogram of the
received signal versus sampling phase and decision threshold, whose open
center area directly predicts link margin).

### 1.2 The gap

The existing options are all closed:

| Option | Cost | Scriptable | Readable | Runs without hardware |
|---|---|---|---|---|
| Bench BERT (Keysight/Anritsu) | $50k–$500k | partially (SCPI) | no | no |
| Intel Transceiver Toolkit | free w/ Quartus | TCL, awkward | no | no |
| Xilinx IBERT | free w/ Vivado | TCL, awkward | no | no |
| **Eyebert** | board cost | Python-native | full RTL + docs | **yes (sim mode)** |

The vendor toolkits work, but they are black boxes. An engineer cannot read
the lock algorithm, cannot extend the register map, cannot put a sweep into
a CI pipeline without fighting a GUI, and a student cannot learn from them at
all. Nothing in the open-source world offers a complete, documented,
verified BERT stack — RTL through plots.

Eyebert's positioning is therefore twofold:

1. **A working tool** for anyone with an Agilex 5 board (and, by porting the
   PHY wrapper, any FPGA with transceivers): scriptable BER sweeps and eye
   capture driven from Python on the embedded Linux core.
2. **A reference and learning vehicle** for everyone else: the simulation
   backend lets the entire toolchain — register accesses, sweep logic,
   statistics, plotting — run on a laptop, and the RTL plus this report
   explain how a real BERT works inside.

### 1.3 Design philosophy

Three principles shaped every decision:

- **Verification before silicon.** Every module has a self-checking
  testbench; the full hierarchy has a system-level testbench with a
  behavioral PHY. Nothing waits for hardware to be tested except the
  analog transceiver itself.
- **No vendor lock below the PHY.** The Native PHY IP and Reconfiguration
  Controller are the only Intel-specific pieces, isolated behind black-box
  wrappers. Everything else is portable SystemVerilog and stdlib-plus-numpy
  Python.
- **Honesty about status.** Simulated results are labeled simulated. The
  bring-up log records what has actually been measured. A portfolio project
  that overclaims is worse than one that under-delivers.

## 2. Background Theory

### 2.1 PRBS sequences and why they work

A pseudo-random binary sequence is generated by a linear feedback shift
register (LFSR). For a *primitive* polynomial of degree N, the LFSR cycles
through every nonzero state exactly once — a maximal-length sequence
(m-sequence) of period 2ᴺ−1. Eyebert implements the three ITU-T O.150
standard patterns:

| Pattern | Polynomial | Period | Longest run of identical bits |
|---|---|---|---|
| PRBS-7 | x⁷ + x⁶ + 1 | 127 | 7 |
| PRBS-15 | x¹⁵ + x¹⁴ + 1 | 32,767 | 15 |
| PRBS-31 | x³¹ + x²⁸ + 1 | ~2.15 × 10⁹ | 31 |

PRBS-31 is the workhorse for stress testing: its long runs of consecutive
identical digits starve the receiver's clock-data recovery (CDR) of
transitions and exercise baseline wander, making it the closest standard
pattern to worst-case real traffic. PRBS-7 is the bring-up pattern: short
period means easy debugging on a scope or in Signal Tap.

**Galois vs Fibonacci form.** The same polynomial can be implemented two
ways. *Fibonacci* (external feedback) XORs several tap bits to produce the
new input bit — a multi-input XOR in the feedback path. *Galois* (internal
feedback) XORs the output bit into multiple state positions — every XOR is
two-input, giving a shorter critical path at high clock rates. The generator
(`prbs_gen.sv`) uses Galois form: on each shift, if the out-going MSB is 1,
the polynomial mask is XORed into the shifted state.

**The recurrence property.** Regardless of implementation form, every
sequence produced by polynomial p(x) = xᴺ + xᴹ + 1 satisfies the linear
recurrence

> b[n] = b[n−(N−M)] ⊕ b[n−N]

This single fact is the foundation of the checker design (§4.2): a receiver
that knows only the last N received bits can predict the next one, with no
knowledge of the transmitter's phase or internal state.

### 2.2 BER statistics: how long must you measure?

Bit errors at a stable operating point are well modeled as a Poisson
process. If the true BER is p and you observe T bits, the expected error
count is λ = pT. Two consequences matter operationally:

**Confidence with zero errors.** Observing zero errors in T bits does *not*
prove BER = 0. At 95% confidence, zero observed errors bounds the true BER
below 3/T (since P(0 errors | λ=3) ≈ e⁻³ ≈ 5%). So:

| Claim | Bits required (95% CL, 0 errors) | Time at 10 Gbps |
|---|---|---|
| BER < 10⁻⁹ | 3 × 10⁹ | 0.3 s |
| BER < 10⁻¹² | 3 × 10¹² | 5 min |
| BER < 10⁻¹⁵ | 3 × 10¹⁵ | 3.5 days |

This is why the counters are **64 bits** (a 32-bit bit counter wraps in
0.43 s at 10 Gbps; 64 bits lasts 58 years) and why `dwell` and
`bits_per_point` are first-class parameters rather than hard-coded.

**Relative error with few counts.** The standard deviation of a Poisson
count is √λ, so measuring BER to ±10% requires λ ≈ 100 errors at the point —
another knob the sweep scripts expose.

### 2.3 The eye diagram

Conceptually, overlay every received unit interval (UI) on top of one
another, triggered by the recovered clock. Clean bits trace two rails and
sharp crossings; jitter smears the crossings horizontally and noise smears
the rails vertically. The empty region in the middle — the *eye opening* —
is where a sampling point can sit and still decide every bit correctly.
**Eye width** (UI fraction) is the horizontal margin against jitter;
**eye height** (voltage fraction) is the vertical margin against noise.

A hardware BERT cannot capture an analog overlay; it rasterizes instead.
The receiver's **phase interpolator** moves the sampling instant in fine
steps across the UI; an offset DAC moves the **decision threshold** across
the voltage range. At each (phase, voltage) point the hardware counts hits
for a fixed dwell, building a 2D histogram — exactly what `eye_sampler.sv`
accumulates in BRAM and `plot_eye.py` renders. The analysis script then
finds the maximal contiguous low-density rectangle and reports it as the
measured eye opening.

### 2.4 Clock domain crossing

The design spans two unrelated clocks: the 100 MHz HPS clock (AXI registers)
and the transceiver recovered clock (all measurement logic). Signals
crossing between them risk **metastability** — a flip-flop sampling a
changing input can hover between logic levels. Three patterns are used,
matched to signal type:

- **Single-bit, level-type** (enables, status flags): a 2-flop synchronizer.
  The first flop may go metastable; it has a full clock period to resolve
  before the second flop samples it. MTBF at these rates is astronomically
  long.
- **Multi-bit, quasi-static** (PRBS mode, sweep geometry, dwell): per-bit
  2FF is only safe because software guarantees the values change while the
  consuming logic is disabled — no consumer ever samples a bus
  mid-transition. This contract is documented and enforced by the software
  layer.
- **Multi-bit, free-running** (the 64-bit counters): per-bit 2FF would
  *tear* — different bits captured from different update cycles. The fix is
  a **req/ack toggle handshake**: a request toggle crosses into the counter
  domain, the counters are latched into snapshot registers, an acknowledge
  toggle crosses back, and only then does the HPS side capture the snapshot.
  The data wires need no synchronizers at all: the protocol guarantees they
  are stable whenever they are sampled.

## 3. System Architecture

```
HPS (Linux, Cortex-A55/A76)
  └─ Python: bert_ctrl.py → /dev/mem mmap @ 0xFF200000
       └─ lightweight HPS-to-FPGA bridge (AXI4-Lite)
            ▼                              hps_clk (100 MHz)
       ┌─────────────┐
       │  axi_csr.sv │   13 registers — docs/register_map.md
       └──────┬──────┘
              │  2FF sync (config →) · toggle handshake (← counters)
══════════════╪══════════════ clock domain boundary ══════════════
              │                       xcvr_rx_clk (recovered)
   ┌──────────┼──────────────┬─────────────────┐
   ▼          ▼              ▼                 │
┌─────────┐ ┌─────────────┐ ┌──────────────┐   │
│prbs_gen │ │ ber_counter │ │ eye_sampler  │   │
│ Galois  │ │ self-sync   │ │ 4096×32 BRAM │   │
│ LFSR ×3 │ │ + LOL       │ │ raster FSM   │   │
└────┬────┘ └──────▲──────┘ └──────▲───────┘   │
     │tx_bit       │rx_bit         │rx_bit     │
     ▼             └───────┬───────┘           │
┌──────────────────────────────────────────────┴───┐
│   Native PHY IP (black-box → Quartus IP)         │◄── Reconfig
│   serializer · TX FFE · CDR · CTLE · DFE · PI    │    Controller
└───┬───────────────────────────────────▲──────────┘
    │ tx_serial_p/n          rx_serial_p/n
    ▼                                   │
   SMA ───────[loopback cable]──────────┘
          XTS-HSMC breakout card
```

Division of labor: **RTL owns time** (bit-exact counting, dwell timing,
raster sequencing — things software cannot do at line rate), **software owns
policy** (which points to sweep, when to reset, where to put the sampling
point via the Reconfiguration Controller, statistics and rendering). The
register map is the entire contract between them.

## 4. Module Design and Key Decisions

### 4.1 prbs_gen — three LFSRs, one mux

Three independent Galois LFSRs (7/15/31 bits) run continuously; `mode`
selects which one drives the output. Costing ~53 flops total, this is
cheaper and cleaner than one reconfigurable LFSR with muxed tap positions,
and it means mode switches take effect in one cycle with all sequences
already warm. Seeds reset to 1 (an all-zero Galois state is a lockup state
that never escapes).

### 4.2 ber_counter — the self-synchronization story

This module went through the project's one major redesign, and the failure
is instructive.

**Version 1 (slip alignment).** The checker instantiated its own copy of
`prbs_gen` and, while hunting, *held* the local LFSR one cycle on each
mismatch — sliding the reference one bit later per slip until 64 consecutive
matches. Module-level tests passed: with generator and checker reset
together, alignment is immediate.

**The failure.** The system-level testbench (`tb_bert_top`) runs the real
topology — PRBS through a 30-bit serial loopback delay — and includes a
loss-of-lock-then-recover scenario. Result: lock was lost correctly, but
**relock never happened**. Root cause: slipping moves the reference in one
direction only. It converges when the reference *leads* the received stream
(the normal cold-start case, where loopback latency puts rx behind), but
after a long corruption event the reference has been held for thousands of
cycles and now *lags* — and the only way forward is to slip through the
entire remaining 2³¹−1 sequence. At 10 Gbps that is about half a second of
dead time in the best case and a simulation eternity; it is simply the wrong
algorithm.

**Version 2 (self-synchronizing).** The shipped design exploits the
recurrence property (§2.1): a 31-bit history register shifts in received
bits, and the recurrence b[n] = b[n−(N−M)] ⊕ b[n−N] predicts each next bit
from history alone. Sixty-four consecutive correct predictions ⇒ lock, from
**any** phase, in N + 64 bits — measured at 65 cycles in `tb_stress`, every
time, regardless of how the streams got misaligned. Once locked, the history
register free-runs on its own predictions so that channel errors cannot
poison the reference. This is how commercial BERT receivers work, and the
project now demonstrates *why*.

**Loss of lock** uses a leaky bucket: +4 per error, −1 per clean bit,
threshold 96. Sustained error density above ~25% (unplugged cable, wrong
data rate) crosses the threshold in tens of bits and drops back to hunt;
isolated errors and short bursts decay harmlessly (an 8-bit burst adds 32 —
well under threshold). Verified end-to-end: corrupt → lock drop in 26 bits →
clean → relock in 65.

### 4.3 eye_sampler — counter-driven raster, software-driven physics

A five-state FSM (IDLE → DWELL → RD → WR → STEP) walks the
`phase_steps × volt_bins` raster, accumulating the count of sampled ones per
dwell window and read-modify-writing it into a 4096 × 32-bit true-dual-port
BRAM (port A: sweep RMW; port B: HPS readback). The RTL deliberately does
*not* touch the analog sampling point — moving the phase interpolator and
threshold DAC requires walking the transceiver's reconfiguration address
map, which is policy, is vendor-specific, and runs at human timescales:
exactly the kind of work that belongs in Python via the Reconfiguration
Controller. Holding `eye_en` high re-rasters continuously, deepening the
histogram for better statistics.

### 4.4 axi_csr — a real AXI4-Lite slave

Independent write-address/write-data acceptance (a master may present them
in either order), registered one-cycle-pulse semantics for `cnt_rst` and
`snap`, byte-strobe support via read-modify-write merging, sentinel
`0xDEADBEEF` on unmapped reads, and no combinational paths between any
ready and valid. The testbench covers strobes, pulse registers, back-to-back
transactions, and reset values.

### 4.5 bert_top — where CDC lives

All synchronizers are inline in the top level, in one place, reviewable at a
glance: 2FF for config and status levels, edge-detected pulse transfer for
`cnt_rst`, the toggle-handshake snapshot for counters, and an
async-assert/sync-deassert reset synchronizer into the recovered clock
domain. The Native PHY and Reconfiguration Controller appear as black-box
wrappers (`ip_stubs.sv` for synthesis elaboration, `phy_bfm.sv` behavioral
models for simulation) so the design compiles and simulates with no vendor
IP present.

### 4.6 The software stack

`csr_map.py` is the single source of truth for offsets and fields — a flat
dictionary mirroring `docs/register_map.md`. `bert_ctrl.py` wraps it in a
context-managed class over a pluggable backend: `DevMem` (mmap of
`/dev/mem`, the real hardware path — no kernel driver needed) or `SimMem`
(§5.3). `sweep.py` orchestrates the three measurement campaigns and saves
compressed `.npz`; the analysis scripts render 300 DPI figures with the eye
script additionally extracting the maximal open rectangle. Dependencies are
deliberately minimal: stdlib + numpy + matplotlib + scipy.

## 5. Verification

### 5.1 Methodology

Layered, all self-checking (`$error`/`$fatal`), all run by `make -C sim all`
and by CI on every push:

| Level | Testbench | What it proves |
|---|---|---|
| Unit | `tb_prbs_gen` | 98,304 bits match independent golden models, all 3 polynomials |
| Unit | `tb_ber_counter` | lock; 10⁶ clean bits, 0 errors; exact counts on injected errors (3 singles → err=3/burst=1; 8-burst → err=11/burst=8) |
| Unit | `tb_eye_sampler` | 4×4 raster, dwell 16, all bins populated, busy semantics |
| Unit | `tb_axi_csr` | reset values, RMW fields, pulse regs, byte strobes, 8× back-to-back, unmapped sentinel |
| Stress | `tb_stress` | PRBS7 period exactly 127; lock from arbitrary phase in 65 cycles; mode-switch relock; cnt_rst atomicity with lock held; counter preservation across en toggle; LOL after 26 corrupt bits → relock in 65; dwell=1; histogram accumulation across sweeps |
| System | `tb_bert_top` | AXI master → CSR → CDC → PRBS → 30-bit serial loopback → checker → snapshot handshake; exact 5-error count through the full stack; LOL/relock through the CSR; eye sweep configured and read back entirely over AXI |
| Software | `pytest` ×8 | field math, no field overlap, RMW isolation, snapshot gating (raw reads frozen between snapshots), monotonic counters, BER-vs-swing monotonicity, eye shape and crossing-vs-center contrast |
| E2E | CI sim sweeps | all three sweep modes + both plot scripts, artifacts uploaded |

### 5.2 Bugs found by the process (kept, deliberately, in this report)

1. **Testbench stimulus race.** Driving `en` with a blocking assignment at
   `posedge` let different module instances sample different values on the
   same edge — the TB generator ran one bit ahead of the checker reference.
   Symptom: PRBS golden mismatch at bit 5; later, an alignment hang. Fix:
   all stimulus drives on `negedge`. Lesson: TB races produce exactly the
   phase offsets that expose checker weaknesses — which led directly to…
2. **Slip alignment cannot recover.** Found only at system level (§4.2).
   The module-level test was green; the topology test was not. This is the
   project's strongest argument for integration testing.
3. **Counter tearing across CDC.** The original design 2FF-synced the
   free-running 64-bit counters — a latent tear documented as a "read twice"
   caveat. Replaced with the snapshot handshake; the pytest suite now
   verifies raw counter reads are frozen between snapshots.
4. **Plot script grabbed the wrong file.** `plot_ber.py` loaded the newest
   `.npz` regardless of mode and died on an eye capture. Both scripts now
   select the newest file *of the matching mode*.

### 5.3 The simulation backend

`sim_backend.py` models the register map and a lossy channel: BER improves
exponentially with TX swing, the CTLE response is a bathtub with an optimum
near mid-range, error counts are Poisson draws against wall-clock-scaled bit
counts, and the synthetic eye histogram's jitter and noise widths track the
same channel quality. It is honest about what it is — a behavioral model for
software development, teaching, and CI — and it means a contributor can
exercise every line of the Python stack, and a student can produce a
physically sensible BER waterfall, on any laptop in under a minute.

### 5.4 Results gallery (simulation mode, labeled as such)

  near gain 10–14, rising four decades toward either extreme.

## 6. Known Limitations (read before extending)

1. **Serial-bit datapath.** The measurement logic consumes one bit per
   recovered-clock cycle. A real Native PHY at multi-Gbps delivers 32/64-bit
   parallel words; Phase 2's main RTL task is widening the PRBS, checker,
   and sampler datapaths (the register map and software are unaffected).
   The architecture was kept bit-serial deliberately — it is the clearest
   form for verification and teaching, and the parallelization is mechanical
   (a parallel LFSR is N applications of the matrix-power trick).
2. **Eye physics is software-driven.** Until the Reconfiguration Controller
   address map for PI/threshold writes is implemented (Phase 4), the eye
   sweep rasters time but not physics; in hardware today every bin would see
   the same sampling point.
3. **Quasi-static CDC is a software contract.** Changing PRBS mode or sweep
   geometry while the engine is enabled is undefined; `BertCtrl` callers
   must (and the provided sweeps do) disable first.
4. **Pin locations are board-file-sourced.** `bert_top.qsf` carries IO
   standards and signal names; physical PIN_ locations import from Terasic's
   baseline QSF, which ships with the board and is not redistributed here.
5. **The sim backend is behavioral, not bit-accurate.** It models the
   register contract and plausible channel statistics, not the RTL. The RTL
   contract is verified by the SystemVerilog testbenches; the two meet at
   the register map.

## 8. Reproducing Everything in This Report

```bash
# RTL: all six testbenches (Icarus Verilog ≥ 12)
make -C sim all

# Software: unit tests
pytest software/tests/

# End-to-end without hardware
cd software/python
python sweep.py --mode ber_ctle --sim --seed 7 --bits 2e8
python sweep.py --mode eye --sim --seed 7
cd ../analysis && python plot_ber.py && python plot_eye.py
```

CI (`.github/workflows/ci.yml`) runs the identical sequence on every push.
